# Chat Backend Implementation

Last updated: 2026-06-30

This document explains the chat backend implemented in this thread. It is intentionally backend-only: API routes, storage, queues, worker jobs, realtime events, environment, migrations, operational checks, and known follow-up work. Frontend details appear only where they affect backend contracts.

## Summary

Chat now has real authenticated backend persistence. It uses hybrid storage:

- Postgres stores durable relationship and inbox metadata.
- AWS Keyspaces stores high-volume message and edit-history rows.
- Upstash Redis stores ephemeral or fast-changing state: unread counters, typing indicators, and presence.
- BullMQ carries delivery/update/read/typing jobs from the API to the worker.
- Ably publishes realtime events from the worker.

The backend is no longer a `501 NOT_IMPLEMENTED` placeholder. API routes exist for inbox, thread creation, message read/write/edit/delete, reactions, read receipts, archive/mute/delete state, typing, and presence. The web remote chat client is aligned to `/v1/chat/inbox`, `/v1/chat/threads`, and `/v1/chat/messages`, and remote mode is the default unless `NEXT_PUBLIC_CHAT_API_MODE=mock` is set for demos/tests.

Frontend cache behavior that affects backend traffic:

- Conversation lists and each opened thread's latest bounded message page are cached in React Query and persisted to `localStorage`.
- Persisted message cache is capped to the most recent 100 messages per thread.
- Infinite/older-history pages are not persisted; older pages still come from `GET /v1/chat/threads/:threadId/messages`.
- The persisted query cache is cleared on sign-out or user switch.
- Chat URLs render lowercase thread IDs for aesthetics, but the frontend normalizes route params back to canonical uppercase thread IDs before API calls.

## Why Hybrid Storage

Chat has two different data shapes:

- Thread membership and inbox previews need relational joins with users, profiles, blocks, mute/archive/delete state, and last-message summaries.
- Message bodies are append-heavy, ordered by thread and time, and can grow much faster than profile/feed metadata.

For that reason, the implementation avoids putting all chat messages in Postgres. Postgres remains the source of truth for thread identity and membership, while Keyspaces handles message timelines.

This split keeps common inbox reads cheap:

- Inbox reads query Postgres metadata and Redis unread counters.
- Message page reads query Keyspaces by `(thread_id, bucket)` and descending `message_id`.
- Message send writes one Keyspaces row plus one Postgres summary upsert.
- Worker jobs hydrate Keyspaces rows only when realtime publishing needs a payload.

The tradeoff is cross-store consistency. A send touches Keyspaces, Postgres, Redis, and BullMQ. The code orders writes so the message row is created before the thread summary and delivery job. If realtime fails, persisted reads still work.

## Storage Model

### Postgres Tables

Postgres stores durable metadata that benefits from relational constraints.

#### `chat_threads`

Purpose: one row per DM or group thread.

Columns:

- `id text primary key`: 35mm ULID-shaped app ID from `createUlid()`.
- `type text not null`: currently `dm` or `group`, validated at API layer.
- `created_by uuid not null`: FK to `users.id`.
- `created_at timestamptz not null default now()`.
- `updated_at timestamptz not null default now()`.

Reasoning:

- Thread ID is application-owned, not generated by Postgres.
- Using text keeps it aligned with other 35mm ULID-like IDs.
- The row is small and stable, so Postgres is appropriate.

#### `chat_participants`

Purpose: active and historical membership for each thread.

Columns:

- `thread_id text not null`: FK to `chat_threads.id`.
- `user_id uuid not null`: FK to `users.id`.
- `joined_at timestamptz not null default now()`.
- `left_at timestamptz`.
- `role text not null default 'member'`.

Constraints and indexes:

- Primary key: `(thread_id, user_id)`.
- Index: `(user_id, joined_at)`.

Reasoning:

- Membership is checked on almost every route.
- The user index supports inbox and membership lookups.
- `left_at` allows soft membership exit without deleting historical thread context.

#### `chat_member_state`

Purpose: per-user state within a thread.

Columns:

- `thread_id text not null`: FK to `chat_threads.id`.
- `user_id uuid not null`: FK to `users.id`.
- `last_read_message_id text`.
- `archived_at timestamptz`.
- `deleted_at timestamptz`.
- `muted_until timestamptz`.

Constraints and indexes:

- Primary key: `(thread_id, user_id)`.
- Index: `user_id`.

Reasoning:

- Archive, mute, delete, and read state are user-specific.
- Keeping state separate from `chat_participants` avoids overloading membership with inbox preferences.
- `deleted_at` means "hidden for this member", not thread deletion.

#### `chat_thread_meta`

Purpose: hot inbox summary.

Columns:

- `thread_id text primary key`: FK to `chat_threads.id`.
- `last_message_at timestamptz`.
- `last_message_preview text`.
- `last_sender_id uuid`: FK to `users.id`, set null on delete.
- `message_count integer not null default 0`.

Indexes:

- `last_message_at`.

Reasoning:

- Inbox must not query Keyspaces per thread just to sort previews.
- Summary rows are updated on send.
- This is denormalized by design.

### AWS Keyspaces Tables

AWS Keyspaces is configured in `us-east-1`:

- Keyspace: `thirtyFiveMM`
- Tables: `messages`, `message_edits`, `message_reactions`
- Capacity mode: on-demand / `PAY_PER_REQUEST`

#### `thirtyFiveMM.messages`

Purpose: primary timeline storage.

Schema:

```cql
CREATE TABLE IF NOT EXISTS thirtyFiveMM.messages (
  thread_id      text,
  bucket         int,
  message_id     timeuuid,
  sender_id      text,
  content_type   text,
  body           text,
  media_url      text,
  media_meta     text,
  link_preview   text,
  reply_to_id    timeuuid,
  reply_snapshot text,
  reactions      map<text, frozen<set<text>>>,
  is_deleted     boolean,
  deleted_at     timestamp,
  edited_at      timestamp,
  created_at     timestamp,
  PRIMARY KEY ((thread_id, bucket), message_id)
) WITH CLUSTERING ORDER BY (message_id DESC)
  AND default_time_to_live = 0;
```

Partition key: `(thread_id, bucket)`.

Clustering key: `message_id DESC`.

Bucket: UTC `YYYYMM`, derived from message time.

Why bucket by month:

- It prevents a high-traffic thread from becoming one unbounded partition.
- It keeps recent message reads efficient.
- The current reader queries the newest 12 monthly buckets.

Why `timeuuid`:

- It naturally encodes creation time.
- It supports descending timeline reads and cursor comparisons.
- It avoids relying on separate created timestamp ordering.

Why `map<text, frozen<set<text>>>` for reactions:

- Reaction state is keyed by emoji, with user IDs as set members.
- AWS Keyspaces rejects nested non-frozen collections, so `map<text,set<text>>` is invalid.
- The implementation uses `frozen<set<text>>` and writes the whole reactions map after reading the existing set. The Node Cassandra driver binds this as a plain object of arrays (`Record<string, string[]>`) so CQL map/set encoding preserves values on later reads.
- This is acceptable only for current low-scale per-message reaction sets. It is not the target shape for viral messages because it causes read-compute-write contention on one message row.

#### `thirtyFiveMM.message_reactions`

Purpose: high-scale reaction fact storage.

Schema:

```cql
CREATE TABLE IF NOT EXISTS thirtyFiveMM.message_reactions (
  thread_id  text,
  bucket     int,
  message_id timeuuid,
  emoji      text,
  shard      tinyint,
  user_id    text,
  reacted_at timestamp,
  PRIMARY KEY ((thread_id, bucket, message_id, emoji, shard), user_id)
);
```

Partition key: `(thread_id, bucket, message_id, emoji, shard)`.

Shard: stable hash of `user_id`, for example `hash(user_id) % 16`.

Why this table exists:

- Adds/removes become single-row writes instead of whole-collection rewrites.
- Hot reaction traffic spreads across shards for one emoji on one message.
- Reads fan out over bounded shards and aggregate in API or async summary state.
- This table should replace `messages.reactions` before production-scale reaction traffic.

#### `thirtyFiveMM.message_edits`

Purpose: edit history.

Schema:

```cql
CREATE TABLE IF NOT EXISTS thirtyFiveMM.message_edits (
  thread_id     text,
  message_id    timeuuid,
  edit_id       timeuuid,
  previous_body text,
  edited_at     timestamp,
  PRIMARY KEY ((thread_id, message_id), edit_id)
) WITH CLUSTERING ORDER BY (edit_id DESC);
```

Reasoning:

- Edits are read by message, not by user or global time.
- Keeping edit history outside the primary message row prevents the main row from accumulating an unbounded list.

### Keyspaces Driver Tuning

The Node.js `cassandra-driver` client is configured in:

- `apps/api/src/lib/keyspaces.ts`
- `apps/worker/src/lib/keyspaces.ts`

Current tuning:

- SigV4 IAM auth on port `9142`.
- `DCAwareRoundRobinPolicy` pinned to `AWS_REGION`.
- `FallthroughRetryPolicy` so ambiguous write errors surface to app code instead of unsafe automatic retries.
- `queryOptions.prepare=true` plus `prepareOnAllHosts=true` and `rePrepareOnUp=true`.
- Execution profiles:
  - `chat-read`: `LOCAL_ONE`, 2s timeout.
  - `chat-write`: `LOCAL_QUORUM`, 2s timeout.
- Warm connection pools at process boot with `warmKeyspacesClient()`.
- Pool env knobs:
  - `KEYSPACES_CORE_CONNECTIONS`
  - `KEYSPACES_MAX_REQUESTS_PER_CONNECTION`
  - `KEYSPACES_CONNECT_TIMEOUT_MS`
  - `KEYSPACES_DEFAULT_TIMEOUT_MS`
  - `KEYSPACES_READ_TIMEOUT_MS`
  - `KEYSPACES_WRITE_TIMEOUT_MS`
  - `KEYSPACES_HEARTBEAT_MS`

## Redis State

Redis usage is intentionally ephemeral and recoverable.

Keys:

- `chat:unread:{userId}:{threadId}`: unread counter.
- `chat:typing:{threadId}:{userId}`: typing marker with 4 second TTL.
- `chat:presence:{userId}`: presence marker with 65 second TTL.

Unread behavior:

- On send, API increments unread counts for every active member except the sender.
- On read receipt, API resets the viewer's unread count for the thread to 0.
- Inbox reads combine Postgres thread summaries with Redis unread counts.

Typing behavior:

- `POST /typing` sets or clears a 4 second marker.
- `GET /typing` scans `chat:typing:{threadId}:*` and returns user IDs.
- The short TTL makes missing explicit "stop typing" calls tolerable.

Presence behavior:

- `POST /presence/ping` sets a 65 second marker for current user.
- `POST /presence/batch` checks markers for up to 50 user IDs.
- Presence is best-effort. If Redis is unavailable, presence returns false and chat persistence still works.

## API Surface

All chat routes are mounted under `/v1/chat` and require `requireAuth`.

### Rate Limits

Current per-user fixed-window limits:

- `chat:inbox`: 60/min
- `chat:create-thread`: 10/min
- `chat:read-messages`: 120/min
- `chat:send-message`: 30/min
- `chat:message-write`: 30/min
- `chat:reaction`: 60/min
- `chat:read-state`: 120/min
- `chat:typing`: 30/min
- `chat:presence`: 4/min

Reasoning:

- Reads are allowed at higher rates.
- Writes, thread creation, and presence pings are more constrained.
- Presence should be client-throttled.

### `GET /v1/chat/inbox`

Query:

- `cursor?: string`
- `limit?: number`, default 20, max 50

Returns:

```ts
ChatInboxPage
```

Behavior:

- Reads threads where current user is active participant.
- Excludes threads where current user's `chat_member_state.deleted_at` is set.
- Sorts by `chat_thread_meta.last_message_at DESC`, then thread ID.
- Uses composite cursor encoding from existing cursor helpers.
- Hydrates members from profiles.
- For DMs, response members exclude the viewer, so the client can render the other participant.
- Adds unread counts from Redis.

Design reason:

- Inbox does not touch Keyspaces.
- This keeps inbox stable even when message storage is under load.

### `POST /v1/chat/threads`

Body:

```json
{
  "type": "dm",
  "memberIds": ["uuid-of-other-user"]
}
```

or:

```json
{
  "type": "group",
  "memberIds": ["uuid-1", "uuid-2"]
}
```

Validation:

- `type` must be `dm` or `group`.
- `memberIds` min 1, max 49.
- DM requires exactly one other member.
- All users must exist.
- Any block relationship between viewer and requested members blocks creation.

DM idempotency:

- If a matching active DM already exists with exactly the same two participants, route returns that thread preview instead of creating a duplicate.

Writes:

- `chat_threads`
- `chat_participants`
- `chat_member_state`
- `chat_thread_meta`

Design reason:

- DM dedupe avoids fragmented conversations.
- Block checks prevent bypassing moderation through chat.
- Group creator becomes `admin`; other members are `member`.

### `GET /v1/chat/threads/:threadId/messages`

Query:

- `before?: string` TIMEUUID message ID
- `limit?: number`, default 50, max 100

Returns:

```ts
ChatMessagesPage
```

Behavior:

- Requires active membership.
- If Keyspaces config is absent, returns empty page for reads. This supports local development where chat message storage may not be configured.
- Queries latest 12 monthly buckets newest-first.
- Uses `message_id < before` for cursor pagination when `before` exists.
- Hydrates sender profile data and reaction summaries.

Important limitation:

- Current reader scans only 12 monthly buckets from "now". Older messages need either archive pagination by bucket, expanded bucket windows, or a thread-level oldest/newest bucket index.

### `POST /v1/chat/threads/:threadId/messages`

Body:

```json
{
  "contentType": "text",
  "body": "hello"
}
```

Supported content types:

- `text`
- `image`
- `gif`
- `file`
- `link`

Validation:

- Message must include `body` or `mediaUrl`.
- `body` max 4000.
- `mediaMetadata` supports width, height, size, mimeType, blurhash.
- `linkPreview` supports URL, title, description, imageUrl, siteName.
- Optional `replyToId` must refer to an existing message in same thread.

Writes:

1. Insert Keyspaces `messages` row.
2. Upsert Postgres `chat_thread_meta`.
3. Increment Redis unread count for every active member except sender.
4. Clear sender typing marker.
5. Publish `message.new` to the thread channel and small-conversation `thread.updated` inbox events directly from the API.
6. Enqueue `chat.deliver` only if direct publish fails or inbox fanout is too large for the API fast path.

Response:

- Hydrated `ChatMessage`.

Design reason:

- Message row is inserted before summary/realtime/job side effects, so persisted message read remains possible even if later side effects fail.
- Reply snapshots store lightweight sender/body/content metadata to avoid needing another Keyspaces lookup to render quoted replies.

### `PATCH /v1/chat/messages/:messageId?threadId=:threadId`

Body:

```json
{
  "body": "edited body"
}
```

Behavior:

- Requires `threadId` query param.
- Derives message bucket from `messageId`.
- Fetches message from Keyspaces.
- Only original sender may edit.
- Deleted messages cannot be edited.
- Inserts previous body into `message_edits`.
- Updates message body and `edited_at`.
- Publishes `message.edited` directly to `thread:{threadId}` after persistence.
- Enqueues `chat.messageUpdated` with type `edit` only when direct publish fails.

Design reason:

- Edit history is durable and separate from the main row.
- Authorization is based on sender ID in message row.

### `DELETE /v1/chat/messages/:messageId?threadId=:threadId`

Behavior:

- Requires `threadId` query param.
- Only original sender may delete.
- Soft-deletes the message in Keyspaces:
  - `is_deleted = true`
  - `deleted_at = now`
  - `body = null`
  - `media_url = null`
- Enqueues `chat.messageUpdated` with type `delete`.

Design reason:

- Message deletion preserves timeline shape, edit/reaction context, and moderation/audit possibility.
- It follows the platform rule that user-generated content should be soft-deleted.

### `POST /v1/chat/messages/:messageId/reactions?threadId=:threadId`

Body:

```json
{
  "emoji": "like"
}
```

Behavior:

- Requires active membership.
- Deleted messages reject reaction add.
- Reads current reaction set for emoji.
- Adds viewer ID.
- Writes the full reactions map back to `messages.reactions`.
- Publishes `message.reaction` directly to `thread:{threadId}` after persistence.
- Enqueues `chat.messageUpdated` with type `reaction` only when direct publish fails.
- Creates a `chat_reaction` notification for the message sender only on the first reaction add by that actor; self-reactions, duplicate retries, inactive recipients, and reaction removals do not create notifications.
- Returns the updated hydrated message so clients can patch the active thread
  cache without a follow-up read.

Why read-compute-write:

- Keyspaces requires nested set values in a map to be frozen.
- Frozen collections cannot be incrementally updated with `+` like non-frozen sets.
- Therefore the route computes the next emoji set and replaces the whole reactions map in one write.

Race note:

- Concurrent reaction writes on the same emoji can race because this is a read-compute-write pattern. For heavy reaction traffic, move reactions to a separate table keyed by `(thread_id, message_id, emoji, user_id)`.

### `DELETE /v1/chat/messages/:messageId/reactions/:emoji?threadId=:threadId`

Behavior:

- Requires active membership.
- Reads current reaction set.
- Removes viewer ID.
- If the set becomes empty, omits that emoji from the next reactions map.
- Writes the full reactions map back to `messages.reactions`.
- Publishes `message.reaction` directly to `thread:{threadId}` after persistence.
- Enqueues `chat.messageUpdated` with type `reaction` only when direct publish fails.
- Returns the updated hydrated message so clients can patch the active thread
  cache without a follow-up read.

### `PATCH /v1/chat/threads/:threadId/read`

Body:

```json
{
  "lastReadMessageId": "timeuuid"
}
```

Behavior:

- Requires active membership.
- Upserts `chat_member_state.last_read_message_id`.
- Resets Redis unread counter to 0.
- Publishes `message.read` directly from the API after read state is stored.
- Enqueues `chat.readReceipt` only if direct publish fails.

Design reason:

- Durable read position lives in Postgres.
- Fast unread badge count lives in Redis.

### `PATCH /v1/chat/threads/:threadId/archive`

Body:

```json
{
  "archived": true
}
```

Behavior:

- Upserts `chat_member_state.archived_at`.
- `archived: false` clears archive state.

### `PATCH /v1/chat/threads/:threadId/mute`

Body:

```json
{
  "mutedUntil": "2026-07-01T00:00:00.000Z"
}
```

Behavior:

- Upserts `chat_member_state.muted_until`.
- Missing or non-string value clears mute state.
- Invalid date returns bad request.

### `DELETE /v1/chat/threads/:threadId`

Behavior:

- Sets `chat_member_state.deleted_at` for current user.
- Does not delete thread globally.
- Inbox excludes deleted threads for that user.

### `POST /v1/chat/threads/:threadId/typing`

Body:

```json
{
  "isTyping": true
}
```

Behavior:

- Requires active membership.
- Sets or clears Redis typing marker.
- Publishes `typing.update` directly from the API after the Redis typing marker is set or cleared.
- Enqueues `chat.typing` only if direct publish fails.

### `GET /v1/chat/threads/:threadId/read-receipts`

Behavior:

- Requires active membership.
- Returns other active participants with `lastReadMessageId` from `chat_member_state`.
- Used by the web active-thread seen indicator as non-polling snapshot state if a `message.read` Ably event was missed.

### `GET /v1/chat/threads/:threadId/typing`

Behavior:

- Requires active membership.
- Returns typing user IDs currently visible in Redis plus profile snippets for the active typing users.
- Used by development/local fallback UI when realtime is not configured. Production live typing uses Ably rather than REST polling.

### `POST /v1/chat/presence/ping`

Behavior:

- Sets viewer presence marker in Redis for 65 seconds.

### `POST /v1/chat/presence/batch`

Body:

```json
{
  "userIds": ["uuid-1", "uuid-2"]
}
```

Behavior:

- Accepts at most 50 IDs.
- Returns map of user ID to boolean online state.

## Worker Jobs and Realtime Events

The API enqueues chat jobs through BullMQ. The worker consumes them in `apps/worker/src/index.ts`.

### `chat.deliver`

Payload:

```ts
{
  messageId: string;
  threadId: string;
  senderId: string;
  bucket: number;
}
```

Worker behavior:

- Fallback/asynchronous path for direct API publish failure or large inbox fanout.
- Fetches message row from Keyspaces.
- Fetches sender profile and active members from Postgres.
- Publishes `message.new` to Ably channel `thread:{threadId}`.
- Publishes `thread.updated` to `user:{memberId}:inbox` for every active member except sender.
- Includes current Redis unread count in inbox event.

Reasoning:

- API does not publish realtime directly. It persists first and delegates fanout to worker.
- This makes message send latency less dependent on Ably and member count.

### `chat.messageUpdated`

Payload:

```ts
{
  messageId: string;
  threadId: string;
  bucket: number;
  type: "edit" | "delete" | "reaction";
}
```

Worker behavior:

- Fetches message row from Keyspaces.
- Hydrates sender profile.
- Publishes to `thread:{threadId}`:
  - `message.edited`
  - `message.deleted`
  - `message.reaction`

Reasoning:

- Update events use the same hydrated `ChatMessage` contract as read responses.
- Clients can update local cache with one event shape.

### `chat.readReceipt`

Payload:

```ts
{
  threadId: string;
  userId: string;
  lastReadMessageId: string;
}
```

Worker behavior:

- Fallback path if API direct `message.read` publish fails.
- Fetches reader username.
- Publishes `message.read` to `thread:{threadId}`.

### `chat.typing`

Payload:

```ts
{
  threadId: string;
  userId: string;
  isTyping: boolean;
}
```

Worker behavior:

- Fallback path if API direct `typing.update` publish fails.
- Fetches user profile snippet.
- Publishes `typing.update` to `thread:{threadId}`.

### Ably Disabled Behavior

If `ABLY_API_KEY` is missing:

- Chat worker jobs log a warning and return false.
- Persistence remains intact.
- Clients using polling/refetch can still see persisted state.

## Contracts

## Consistency and Failure Modes

### Send Path

Send path writes multiple systems:

1. Keyspaces message row.
2. Postgres thread summary.
3. Redis unread counters.
4. BullMQ delivery job.

If Keyspaces fails:

- Send returns `503 KEYSPACES_UNAVAILABLE` or driver error.
- No message is persisted.

If Postgres summary fails after Keyspaces insert:

- Message row may exist without inbox summary update.
- This would require repair by reading latest message from Keyspaces and updating `chat_thread_meta`.

If Redis unread increment fails:

- Message persists.
- Unread badge may be stale until read state or later repair.

If BullMQ enqueue fails:

- Message persists.
- Realtime event may not fire.
- Client refetch still sees message.

### Read Path

Inbox reads:

- Depend on Postgres and Redis.
- Do not depend on Keyspaces.

Message reads:

- Depend on Keyspaces.
- If Keyspaces client config is missing, route returns empty page for local development.
- If Keyspaces is configured but fails at query time, error propagates.

### Reactions

Reactions use read-compute-write for the frozen-set-backed reactions map. This is simple and Keyspaces-compatible but not contention-proof. Heavy concurrent reaction traffic can drop one writer's update. Future hardening should use a separate reaction fact table.

### Message Deletion

Deletion is soft. The message row remains so clients can preserve order and show a deleted placeholder.

## Security and Authorization

All routes require authenticated user context from Clerk via `requireAuth`.

Membership:

- Message reads/writes, typing, and read state require active membership.
- `left_at` excludes a participant from active membership.

Sender-only actions:

- Only sender can edit a message.
- Only sender can delete a message.

Thread creation:

- All requested users must exist.
- Existing block relationships between viewer and invitees block thread creation.

Known gap:

- Group admin semantics are minimal. Creator is assigned `admin`, but no admin-only management routes exist yet.

## Troubleshooting

### `KEYSPACES_UNAVAILABLE`

Likely causes:

- Missing `AWS_REGION` and `KEYSPACES_ENDPOINT`.
- Missing IAM access key env vars.
- Wrong region or endpoint.
- IAM policy does not allow table access.
- Keyspaces table not active.

### `Non-frozen collections are not allowed inside collections`

Cause:

- Attempted `map<text,set<text>>`.

Fix:

- Use `map<text,frozen<set<text>>>`.
- Update reactions by replacing one map entry with the full set.

### `does not provide an export named 'AmazonKeyspacesSigV4AuthProvider'`

Cause:

- Installed `aws-sigv4-auth-cassandra-plugin` exports `SigV4AuthProvider`.

Fix:

- Import default module and instantiate `new sigV4.SigV4AuthProvider({ region })`.

### `drizzle-kit migrate` sees `DATABASE_URL` as undefined

Cause:

- Drizzle config reads `process.env.DATABASE_URL`.
- Shell `source apps/api/.env` may fail when URL contains `&`.

Fix:

- Use Node env file:

```bash
node --env-file=../../apps/api/.env ../../node_modules/drizzle-kit/bin.cjs migrate
```

### Realtime Not Arriving

Check:

- `ABLY_API_KEY` exists in worker env.
- Worker is running.
- BullMQ Redis URL is configured.
- Job names are dispatched in `apps/worker/src/index.ts`.
- Client subscribes to the matching channel:
  - `thread:{threadId}`
  - `user:{userId}:inbox`

Persistence does not depend on Ably.

## Design Decisions To Preserve

- Keep thread and membership metadata in Postgres.
- Keep messages in Keyspaces with `(thread_id, bucket)` partitioning.
- Keep message IDs as TIMEUUID strings in API payloads.
- Do not use offset pagination.
- Keep thread inbox reads independent of Keyspaces.
- Keep user delete/archive/mute state per member, not global.
- Soft-delete messages instead of hard deleting.
- Treat Redis and Ably as best-effort layers; persistence must not require realtime success.
- Keep AWS Keyspaces auth on IAM SigV4 access keys, not service-specific Keyspaces username/password credentials.
